# Guest Sign-In Test Suite - Implementation Analysis

## Executive Summary

A comprehensive Playwright test suite has been implemented to verify guest sign-in functionality in the Inspire webapp. The suite includes:

- **2 Test Cases** covering full sign-in flow and username format validation
- **13 Screenshot Checkpoints** for visual verification at each stage
- **Screenshot Helper Utilities** for automated capture and analysis
- **Detailed Logging** for debugging and audit trails
- **MCP Image Analysis Integration** for advanced screenshot analysis

## Test Cases Breakdown

### Test 1: Full Guest Sign-In Flow

**File**: `frontend/tests/guest-signin.spec.ts` (lines 11-112)

**Test Name**: "should sign in as guest and verify username appears in top left"

**Workflow**:

```
1. Navigate to homepage
   └─ Screenshot: 01-homepage-initial
   └─ Verify page load with networkidle

2. Check navigation state
   └─ Screenshot: 02-check-nav-handle
   └─ Get page accessibility info (headings, buttons)

3. Detect if authentication required
   └─ Check .nav-handle element visibility
   └─ Compare text for auth indicators

4. Open authentication modal
   └─ Click auth trigger button
   └─ Screenshot: 03-auth-trigger-visible
   └─ Wait for modal overlay

5. Select Guest Mode
   └─ Screenshot: 04-auth-modal-open
   └─ Click Guest Mode tab
   └─ Screenshot: 05-guest-mode-tab-active
   └─ Verify guest description text

6. Complete guest authentication
   └─ Click "Continue as Guest" button
   └─ Screenshot: 06-before-guest-auth
   └─ Wait for modal to close

7. Verify username display
   └─ Check .nav-handle is visible
   └─ Screenshot: 07-after-auth-nav-visible
   └─ Extract username text

8. Validate username
   └─ Ensure non-empty
   └─ Verify not default auth text
   └─ Screenshot: 08-final-success-state
   └─ Final verification
```

**Assertions**:
- `navHandle` is visible after auth
- Username is non-empty (length > 0)
- Username doesn't contain "Sign up" or "Log in"
- Screenshot captures successful state

**Success Criteria**:
✅ Modal opens when auth button clicked
✅ Guest Mode tab is selectable
✅ Authentication completes without errors
✅ Username appears in top-left navigation
✅ All 8 screenshots captured successfully

### Test 2: Guest Username Format Validation

**File**: `frontend/tests/guest-signin.spec.ts` (lines 114-179)

**Test Name**: "should display guest username format correctly"

**Workflow**:

```
1. Load homepage
   └─ Screenshot: 01-format-test-homepage

2. Check authentication status
   └─ Screenshot: 02-format-check-auth-status
   └─ Determine if sign-in needed

3. If sign-in required:
   └─ Open modal
   └─ Screenshot: 03-format-auth-modal-open
   └─ Select guest mode
   └─ Screenshot: 04-format-guest-tab-active
   └─ Complete authentication

4. Get username
   └─ Screenshot: 05-format-username-visible
   └─ Extract from .nav-handle

5. Validate format
   └─ Check length > 5
   └─ Verify no @ symbol
   └─ Match regex: /^[A-Z][a-zA-Z]+\d{1,4}$/
   └─ Log result
```

**Format Rules**:

Guest usernames are generated by backend (`backend/src/auth/store.ts`) using:

```
[Adjective] + [Noun] + [4-digit number]
```

Examples:
- `CoolArtist1234`
- `EpicWizard42`
- `SuperGenius999`
- `FreshCreator88`

**Assertions**:
- Username length > 5
- No `@` symbol present
- Matches pattern: `[A-Z][a-zA-Z]+\d{1,4}`

**Success Criteria**:
✅ Username generated successfully
✅ Format matches expected pattern
✅ No reserved characters present
✅ All 5 screenshots captured

## Screenshot Capture Implementation

### Screenshot Helper Module

**Location**: `frontend/tests/helpers/screenshot-helper.ts`

**Key Functions**:

#### 1. `captureScreenshot(page, name, elementSelector?)`

Captures full-page screenshot with element tracking.

```typescript
// Example usage:
const result = await captureScreenshot(page, 'auth-modal-open', '.auth-modal');

// Returns:
{
  path: 'test-artifacts/auth-modal-open-1706804523456.png',
  timestamp: 1706804523456,
  elementFound: true,
  elementText: 'Welcome to Inspire',
  elementBounds: { x: 100, y: 50, width: 500, height: 300 }
}
```

#### 2. `logScreenshotStage(page, stageName, elementSelector?)`

Captures and logs detailed information.

**Output**:
```
📸 Screenshot: 05-guest-mode-tab-active
   Path: test-artifacts/05-guest-mode-tab-active-1706804524890.png
   Time: 2024-02-01T15:35:24.890Z
   Element found: true
   Element text: Continue as a guest...
   Element bounds: 150,200 (250x50)
```

#### 3. `getPageAccessibility(page)`

Retrieves page metadata for debugging.

```typescript
{
  url: 'http://localhost:8080/',
  title: 'Inspire - Make Something',
  bodyClasses: 'app theme-default mode-landing',
  headingsCount: 8,
  buttonsCount: 24
}
```

#### 4. `verifyElementWithScreenshot(page, selector, screenshotName, expectVisible?)`

Visual verification with screenshot.

```typescript
const isVisible = await verifyElementWithScreenshot(
  page,
  '.nav-handle',
  'nav-handle-check',
  true // expect visible
);
```

#### 5. `waitAndCapture(page, selector, screenshotName, timeout?)`

Waits for element and captures.

```typescript
const result = await waitAndCapture(
  page,
  '.auth-guest-btn',
  'guest-btn-appeared',
  5000
);
```

### Screenshot Organization

Screenshots follow naming convention:

```
{stage}-{timestamp}.png

Examples:
- 01-homepage-initial-1706804523456.png
- 02-check-nav-handle-1706804524123.png
- 03-auth-trigger-visible-1706804524567.png
```

**Benefits**:
- Chronological ordering (numbered prefix)
- Unique identification (timestamp)
- Human-readable stage names
- Easy sorting and grouping

## Image Analysis Integration

### MCP Tools Available

The test suite integrates with Model Context Protocol (MCP) image analysis tools:

#### 1. Object Detection (`mcp_sunriseapps_i_detect`)

Detects UI elements in screenshots:

```bash
mcp_sunriseapps_i_detect \
  --input_path test-artifacts/04-auth-modal-open-*.png \
  --return_geometry mask
```

**Use Cases**:
- Find login buttons
- Detect text input fields
- Identify modal boundaries
- Locate navigation elements

#### 2. Optical Character Recognition (`mcp_sunriseapps_i_ocr`)

Extracts text from screenshots:

```bash
mcp_sunriseapps_i_ocr \
  --input_path test-artifacts/07-after-auth-nav-visible-*.png \
  --language en
```

**Use Cases**:
- Verify username text
- Check error messages
- Validate button labels
- Extract form content

#### 3. Object Finding (`mcp_sunriseapps_i_find`)

Finds objects by description:

```bash
mcp_sunriseapps_i_find \
  --input_path test-artifacts/04-auth-modal-open-*.png \
  --description "Guest Mode tab" \
  --return_geometry polygon
```

**Use Cases**:
- Locate specific UI elements
- Find interactive components
- Identify visual states

## Running the Tests

```bash

cd frontend
npx playwright test guest-signin.spec.ts

npx playwright test guest-signin.spec.ts --headed

npx playwright test guest-signin.spec.ts --verbose
```

```bash
node run-guest-signin-tests.js --headed
```

## Known Issues & Workarounds

### Issue 1: HTTPS Server Connection Failure

**Symptom**: `net::ERR_CONNECTION_REFUSED` on HTTPS tests

**Root Cause**: Vite server not binding to specified IP

**Workaround**: Use standard HTTP localhost config

```bash
# Use default config (HTTP localhost)
npx playwright test guest-signin.spec.ts

# Not HTTPS IP
# npx playwright test --config=playwright.config.https.ts
```

### Issue 2: Element Not Found

**Symptom**: `.nav-handle` or `.auth-modal-overlay` not found

**Root Cause**: Selector name changed or element not rendered

**Check**:
```bash
# Run headed to visually inspect
npx playwright test guest-signin.spec.ts --headed

# Use debug mode
npx playwright test guest-signin.spec.ts --debug
```

### Issue 3: Screenshot Directory Not Created

**Symptom**: Screenshots not saved

**Root Cause**: Directory doesn't exist

**Fix**:
```bash
mkdir -p test-artifacts
```

## Maintenance Guidelines

### Updating Selectors

If HTML structure changes:

```typescript
// OLD:
const navHandle = page.locator('.nav-handle');

// NEW (if class name changes):
const navHandle = page.locator('[aria-label="username"]');
```

### Adding New Tests

Template:

```typescript
test('new test case', async ({ page }) => {
  await page.goto('/');
  await logScreenshotStage(page, '01-stage-name');
  
  // Your test logic
  
  await logScreenshotStage(page, '02-final-state', '.element');
  expect(true).toBe(true); // Your assertions
});
```

### Analyzing Test Failures

1. Check console output for errors
2. Review screenshots in order
3. Use MCP tools on specific screenshots
4. Run in headed mode for live debugging
5. Check browser DevTools (--debug mode)
